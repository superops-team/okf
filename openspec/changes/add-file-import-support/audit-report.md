# 全维度问题校验报告

## 1. 上下文逻辑连贯性校验

### 问题识别
- **一致性问题**：proposal.md 中提到 `Config`、`ConfigManager` 两个模型，但 design.md 只定义了 `Config` 结构，缺少 `ConfigManager` 的设计
- **API 命名不一致**：design.md 中 `Import()` 函数描述为"Auto-detect and import"，但未说明与 `ImportFile/ImportDirectory/ImportArchive` 的调用关系

### 建议修正
- 统一术语：明确 `ConfigManager` 是 `Config` 的管理逻辑，或直接合并到 `Config` 相关函数中
- 补充 API 调用关系说明：`Import()` 作为入口函数，内部根据文件类型自动路由到对应的导入函数

## 2. 内容空洞排查

### 问题识别
- **"知识管理能力"描述模糊**：proposal.md 提到"配置管理、路径查询、状态查看"，但未明确"状态查看"的具体内容
- **缺乏落地依据**：design.md 中提到"memory usage MUST not exceed 512MB"，但未说明如何监控和限制内存使用

### 建议修正
- 明确"状态查看"的具体内容：知识库统计信息（概念数量、文件数、最后更新时间等）
- 移除或明确性能指标的落地方式：如需内存限制，应设计具体的实现策略（如分批处理、流式读取）

## 3. 歧义点识别

### 问题识别
- **"preserve original filename and directory structure"**：spec.md 第42行，未说明是保留完整绝对路径还是相对路径
- **"import all .md files"**：spec.md 第49行，未明确是否递归子目录（虽然后续提到了递归，但表述不一致）

### 建议修正

- CLI 帮助信息中明确默认值

## 5. SDD/TDD 适配评估

### 潜在阻碍
| 维度 | 问题 | 影响 |
|------|------|------|
| **单元测试隔离** | `ResolveKnowledgeDir` 依赖环境变量和平台检测 | 难以在单一平台测试多平台行为 |
| **测试数据准备** | 压缩包导入需要真实归档文件 | 测试依赖外部文件，增加维护成本 |
| **性能测试** | 1000文件导入的性能指标难以在单元测试中验证 | 需要单独的性能测试套件 |

### 适配方案
- 使用 mock/stub 替代真实环境变量
- 内嵌测试用的小型归档文件（base64编码或生成）
- 将性能测试分离为独立测试组

---

## 6. 最小化实现原则检查

### 问题识别
- **过度设计风险**：`ImportOptions` 中的 `Recursive` 选项是否必要？默认递归即可，无需额外配置
- **边界模糊**：未明确导入文件的大小限制、深度限制
- **可测试性**：`Import()` 函数承担了类型判断和导入逻辑，职责过重

### 建议修正
- 移除 `Recursive` 选项：默认递归，如需非递归可通过其他方式实现
- 明确边界：单文件最大大小、目录最大深度
- 拆分 `Import()`：类型检测与实际导入分离，提高可测试性

---

## 7. 向下兼容隐患

### 问题识别
- **配置文件格式**：新增 config 文件可能与现有 `.okf/` 目录结构冲突
- **路径变更**：默认路径从当前目录 `.okf/knowledge` 变为用户目录，可能影响现有项目

### 建议修正
- 配置文件放置位置：`~/.okf/config.yaml`，与知识库目录分离
- 向后兼容策略：若当前目录存在 `.okf/knowledge`，优先使用；否则使用平台默认

## 9. 风险点预判

### 高风险点
1. **路径遍历攻击**：压缩包提取时的路径遍历（已在 spec 中提及，但实现需严格验证）
2. **资源耗尽**：超大压缩包或深层目录结构导致内存溢出
3. **文件权限问题**：导入时权限设置不当导致后续读取失败
4. **并发安全**：配置文件读写的并发访问（虽当前为单进程，但需考虑未来扩展）

### 建议措施
- 强制路径规范化：`filepath.Clean()` + 前缀检查
- 限制解压深度和文件数量
- 统一权限设置：文件 0644，目录 0755
- 配置文件使用原子写入

## 12. 可扩展性评估

### 当前方案的扩展性

**优点**：
- 配置结构可扩展（预留 `// ... other config options`）
- 导入选项设计为结构体，便于添加新选项
- 模块划分清晰，职责单一

**潜在问题**：
- `ImportResult` 结构固定，新增统计项需修改结构
- 压缩包处理耦合在 `import.go`，新增格式需修改核心逻辑

### 改进建议
- 使用 `map[string]interface{}` 或 protobuf 作为扩展字段容器
- 将压缩格式处理抽象为接口，便于扩展

---

## 13. 过度设计检查

### 当前设计评估
- **合理部分**：配置管理、导入选项、结果统计
- **可能过度**：`ConfigManager` 单独模型（可简化为函数）、`ImportOptions.Recursive`（默认递归即可）

### 简化建议
- 移除 `ConfigManager`，将配置管理逻辑作为 `Config` 的方法或独立函数
- 移除 `Recursive` 选项，默认递归导入
- 压缩包格式检测使用简单的文件扩展名判断，而非复杂的魔术字节检测

---

## 14. 最小改动原则

### 当前方案评估
- **符合原则**：不修改现有核心类型（Concept、KnowledgeBundle）
- **可优化**：复用现有 parser 模块进行 OKF 验证，避免重复实现

### 优化建议
- 直接调用 `pkg/parser` 的验证函数
- CLI 命令复用现有命令框架和错误处理模式

---

## 15. 代码优雅性评估

### 当前设计评估
- **优点**：API 设计简洁，职责划分清晰
- **待改进**：错误处理策略不够明确，缺少统一的错误类型

### 优化建议
- 定义统一的错误类型（如 `ImportError`）
- 提供详细的错误信息和恢复建议

---

## 16. 架构一致性检查

### 当前方案评估
- **符合项目风格**：遵循现有模块划分（pkg/okf, cmd/okf）
- **潜在割裂**：新增配置管理可能与现有 `pkg/okf/api.go` 产生职责重叠

### 优化建议
- 将配置逻辑整合到 `pkg/okf/api.go` 或创建独立的 `pkg/okf/config.go`
- CLI 命令风格与现有命令保持一致（如 `okf init`, `okf lint`）

## 综合评估结论

| 维度 | 评级 | 说明 |
|------|------|------|
| 整体可行性 | **高** | 技术方案成熟，风险可控 |
| 实现复杂度 | **中** | 需要处理较多边界情况 |
| 向后兼容性 | **中** | 需要兼容现有知识库路径 |
| 可测试性 | **高** | 模块划分清晰，便于单元测试 |
| 可扩展性 | **中** | 基础架构良好，部分设计可优化 |

### 核心改进建议

1. **简化配置模型**：移除 `ConfigManager`，使用简单函数实现配置管理
2. **明确路径语义**：定义清楚相对路径 vs 绝对路径的处理规则
3. **强化安全措施**：完善路径遍历防护、资源限制
4. **优化测试策略**：使用 mock 和内嵌测试数据提高测试可维护性
5. **增强错误处理**：定义统一错误类型和处理策略
