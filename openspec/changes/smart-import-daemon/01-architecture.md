## 2. 核心组件职责

### 2.1 Metadata Manager（元数据管理器）

**职责**：
- 维护所有已导入文件的源路径、目标路径、内容哈希、时间戳、源文件大小
- 提供查询、更新、删除元数据接口
- 支持原子化读写，保证并发安全

**存储格式**：`knowledge/.metadata.json`

```json
{
  "version": "1.0",
  "last_synced": "2024-01-15T10:30:00Z",
  "files": {
    "api/user.md": {
      "source_path": "/home/user/docs/api/user.md",
      "content_hash": "a1b2c3d4e5f6...",
      "frontmatter_hash": "x7y8z9...",
      "last_modified": "2024-01-15T09:00:00Z",
      "last_imported": "2024-01-15T10:30:00Z",
      "file_size": 1024,
      "import_strategy": "merge"
    }
  }
}
```

### 2.2 Hash&Stat Detector（检测器）

**职责**：
- 检测源文件是否发生变更（快速路径）
- 先通过 `mtime + size` 快速检查（O(1) 系统调用）
- 若疑似变更，再通过内容 hash 精准确认（IO 读取）

**检测算法**：
```
1. 读取元数据中的记录
2. 检查源文件 mtime 和 size
   ├─ 与记录一致 → 无变更，直接跳过 (FAST PATH)
   └─ 与记录不同 → 计算内容 hash (SLOW PATH)
3. 检查 hash
   ├─ 与记录一致 → 更新 mtime/size，无实际变更 (hash collision safe)
   └─ 与记录不同 → 有实际变更，交给 Merge Engine
```

### 2.3 Merge Engine（合并引擎）

**职责**：
- 决定文件变更处理策略（四种）
- 实现每种策略的核心逻辑
- 提供策略决策树

**四种策略**：

| 策略 | 行为 | 适用场景 |
|------|------|---------|
| `skip` (默认) | 源文件和目标文件都存在时跳过 | 保守导入，防止覆盖 |
| `overwrite` | 直接用源文件覆盖目标 | 源文件为唯一真源 |
| `merge` | 合并 frontmatter，保留目标文件的自定义字段 | 目标文件有自定义标签 |
| `patch` | 仅更新 frontmatter 中指定字段，不覆盖整个文件 | 精细更新 |

### 2.4 Watch Daemon（守护进程）

**职责**：
- 使用 `fsnotify` 监听文件系统事件
- 事件去抖和聚合（避免编辑器保存时多次触发）
- 按 Watch 规则过滤事件
- 调度 Import Engine 执行更新

**事件处理流程**：
```
文件系统事件 (CREATE/WRITE/RENAME/REMOVE)
       │
       ▼
  ┌────────────┐
  │ Debouncer  │ - 500ms 冷却窗口 - 同一文件在窗口内的多次事件合并
  └─────┬──────┘
       │
       ▼
  ┌────────────┐
  │ 规则匹配器 │ - 检查是否匹配 Watch 规则
  └─────┬──────┘
       │
       ▼
  ┌────────────┐
  │ 任务队列   │ - 串行处理，避免并发导入
  └─────┬──────┘
       │
       ▼
  ┌────────────┐
  │ Import     │ - 调用 Smart Import Engine
  │ Engine     │
  └────────────┘
```

## 4. 数据流

### 4.1 单次导入流程

```
用户执行 okf add
    │
    ▼
1. 读取目标知识库的 .metadata.json（不存在则初始化空索引）
    │
    ▼
2. 遍历源文件
    ├─ 对每个源文件:
    │   ├─ 根据源路径查询元数据
    │   │   ├─ 有记录 → 进入 Hash&Stat 检测
    │   │   └─ 无记录 → 新文件，正常导入
    │   ├─ Hash&Stat 检测
    │   │   ├─ FAST PATH: mtime+size 相同 → skip (几乎无IO)
    │   │   └─ SLOW PATH: mtime+size 不同 → 计算 hash
    │   │       ├─ hash 相同 → 更新元数据中 mtime/size，skip (无内容变更)
    │   │       └─ hash 不同 → 交给 Merge Engine
    │   └─ Merge Engine 处理
    │       ├─ 检查文件中 frontmatter 的 x-import-strategy 字段
    │       ├─ 或使用用户通过 CLI 传入的策略
    │       └─ 执行对应策略
    │
    ▼
3. 原子化写入 .metadata.json（先写 .tmp，再 rename）
    │
    ▼
4. 返回 ImportResult（含导入/跳过/失败统计）
```

### 4.2 Watch Daemon 流程

```
用户执行 okf watch --dir ./knowledge
    │
    ▼
1. 读取 .watch.yaml 规则配置
    │
    ▼
2. 初始化 fsnotify.Watcher，添加规则中的监听路径
    │
    ▼
3. 启动事件循环
    ├─ 接收 fsnotify 事件
    ├─ 应用 Debouncer (500ms 窗口)
    ├─ 匹配 Watch 规则
    ├─ 排入任务队列
    └─ 执行 Smart Import (同 4.1 流程)
    │
    ▼
4. 监听信号 (SIGINT/SIGTERM/SIGHUP)
    ├─ SIGINT/SIGTERM → 优雅退出
    └─ SIGHUP → 重新加载 .watch.yaml 规则
```
