## Context

OKF 的基础抽象是 Markdown-backed `Concept` 和目录级 `KnowledgeBundle`。`Concept` 已支持 `Type`、`Title`、`Description`、`Resource`、`Tags`、`Content` 和 `CustomFields`（`/Users/bytedance/workspace/github/okf/pkg/okf/types.go:11`），`KnowledgeBundle` 是概念集合和分发单位（`/Users/bytedance/workspace/github/okf/pkg/okf/types.go:51`）。这意味着代码知识扩展应该优先复用现有 Concept/Bundle，而不是新增并行存储。

CodeGraph 的模型更接近代码结构数据库：`NODE_KINDS` 覆盖 file/module/class/struct/interface/function/method/import/component 等节点类型（`/Users/bytedance/workspace/github/codegraph/src/types.ts:18`），`EdgeKind` 覆盖 contains/calls/imports/extends/implements/references/type_of/returns 等关系（`/Users/bytedance/workspace/github/codegraph/src/types.ts:48`）。`Node` 包含 stable-ish id、kind、name、qualifiedName、filePath、language、line/column range、docstring、signature、visibility、exported flags（`/Users/bytedance/workspace/github/codegraph/src/types.ts:110`）。`Edge` 包含 source、target、kind、metadata、line/column、provenance（`/Users/bytedance/workspace/github/codegraph/src/types.ts:184`）。SQLite schema 对应持久化 nodes、edges、files 和 FTS indexes（`/Users/bytedance/workspace/github/codegraph/src/db/schema.sql:20`、`/Users/bytedance/workspace/github/codegraph/src/db/schema.sql:45`、`/Users/bytedance/workspace/github/codegraph/src/db/schema.sql:97`）。

本 change 的设计目标是让 OKF 成为“代码知识发布层”：它能从自身轻量 analyzer 或外部 CodeGraph-like extractor 接收实体/关系，并渲染为可读、可 diff、可增量更新的 Markdown 知识库。

## Goals / Non-Goals

**Goals:**

- 定义代码维度核心模型：repository、file、package/module、symbol、relation、context view。
- 定义 CodeGraph Node/Edge/FileRecord 到 OKF Concept/metadata/Markdown 的稳定映射。
- 支持生成仓库级索引页面和关系页面，帮助 Agent 从“任务/符号/文件”快速找到相关上下文。
- 与现有增量状态兼容：文件变化后实体、关系和 derived views 都能收敛。
- 默认不复制完整源码，只记录 path、line range、signature、docstring、摘要和关系。

**Non-Goals:**

- 不在本 change 强制接入 tree-sitter、CodeGraph npm 包、SQLite 或 MCP server。
- 不要求实现精确 callers/callees/SSA/LSP 级调用图；若外部 CodeGraph 提供 calls 边，OKF 可导入并保留 provenance。
- 不改变 OKF 基础 frontmatter 必填字段。
- 不把 OKF 变成长期运行 daemon 或数据库服务。

## Model

### CodeEntity

`CodeEntity` 是所有代码维度节点的规范化内部模型。

Required fields:

- `id`: deterministic identity, generated by OKF rather than trusting extractor-specific ids.
- `kind`: `repository | file | package | module | symbol | import | context_view` plus symbol sub-kinds.
- `name`: short display name.
- `qualified_name`: language-aware qualified name when available.
- `language`: known language or `unknown`.
- `file_path`: repository-relative path when applicable.
- `start_line` / `end_line`: 1-indexed line range when applicable.
- `source`: `okf-go-ast | okf-regex | codegraph | manual`.
- `schema_version`: code dimension schema version.

Recommended optional fields:

- `signature`, `docstring`, `visibility`, `exported`, `receiver`, `package`, `content_hash`, `errors`, `metadata`.

Stable identity format:

```text
code:<repo-id>:<file_path>#<kind>:<qualified_name>@L<start_line>-L<end_line>
```

For file/package/repository entities without line ranges, omit the `@L...` suffix. The identity MUST be deterministic across full regeneration and incremental update for unchanged source structure.

### CodeRelation

`CodeRelation` describes directed relationships.

Required fields:

- `source_id`
- `target_id` or `target_ref` for unresolved imports/references
- `kind`: normalized relation kind
- `provenance`: `ast | regex | codegraph | heuristic | manual`
- `file_path`, `line`, `column` when a source location exists

Normalized relation kinds include:

- `contains`: repository→package, package→file, file→symbol, type→method
- `imports`: file/package→import target
- `calls`: function/method→function/method if available
- `extends`, `implements`, `references`, `type_of`, `returns`: retained when provided by a richer extractor
- `documents`: concept→code entity when a human-written concept explains code

### Generated Concept Types

The first implementation SHOULD keep file-level concepts as the default user-facing artifact and add generated index concepts for repository-level navigation:

- `code_repository`: repository overview, languages, package/module map, entrypoints, generated-view manifest.
- `code_file`: one generated concept per indexed source file.
- `code_package`: optional package/module summary for languages where package/module can be detected.
- `code_symbol`: optional standalone concept for important exported symbols or configured symbol kinds.
- `code_relation_index`: compact machine-readable and human-readable relation tables.
- `code_context_view`: generated slices such as entrypoints, recently changed symbols, public API, test surface, dependency boundaries.

## CodeGraph Compatibility Mapping

| CodeGraph model | OKF code dimension mapping |
| --- | --- |
| `Node.kind=file` | `CodeEntity{kind:file}` and `code_file` concept |
| `Node.kind=module/namespace` | `CodeEntity{kind:module}` or `code_package` concept |
| `Node.kind=function/method/class/struct/interface/type_alias/enum/component` | `CodeEntity{kind:symbol, symbol_kind:<node.kind>}`; embedded in `code_file` and optionally standalone `code_symbol` |
| `Node.qualifiedName` | `qualified_name`, used in stable identity |
| `Node.filePath/startLine/endLine/startColumn/endColumn` | navigation metadata and `file_path:start_line-end_line` display |
| `Node.docstring/signature/visibility/isExported` | frontmatter custom fields plus Markdown symbol table |
| `Edge.kind` | `CodeRelation.kind` with same value when supported |
| `Edge.metadata/provenance/line/column` | relation metadata and provenance |
| `FileRecord.contentHash/language/errors` | file entity metadata and incremental freshness fields |
| `Subgraph` | generated `code_context_view` or query result view |
| `Context` | generated or on-demand Agent context bundle, not necessarily persisted |

Important compatibility decision: OKF MUST NOT persist CodeGraph's raw node id as the primary identity. It MAY store it as `external_ids.codegraph`, but OKF identity is derived from repository-relative source metadata so it remains stable if the extractor implementation changes.

## Data Flow

1. **Extract**: Use existing OKF analyzers for Go AST and fallback languages; optionally accept CodeGraph-like JSON or future direct CodeGraph DB import.
2. **Normalize**: Convert extractor-specific records into `CodeEntity` and `CodeRelation`.
3. **Reconcile**: Use stable identity and file path to upsert changed entities, remove stale generated entities for deleted files, and preserve human-written concepts.
4. **Render**: Write Markdown concepts with stable sections: summary, source location, package/module, imports, symbols, relationships, provenance, warnings.
5. **Index**: Build in-memory query indexes for type/tag/resource/symbol/relation dimensions.
6. **View**: Generate repository-level navigation concepts and context views.

## Markdown and Metadata Shape

Generated concepts SHOULD use existing OKF frontmatter fields plus code-specific custom fields.

Example frontmatter shape:

```yaml
type: code_file
title: src/index.ts
resource: code://repo/src/index.ts
tags: [code, typescript, generated]
code:
  schema_version: 1
  entity_id: code:repo:src/index.ts#file:src/index.ts
  language: typescript
  file_path: src/index.ts
  content_hash: abc123
  generated: true
```

Example body sections:

```markdown
## Source
- Path: `src/index.ts`
- Language: `typescript`
- Lines: 1-260

## Symbols
| Kind | Name | Signature | Location | Exported |
| --- | --- | --- | --- | --- |

## Relationships
| Kind | Target | Location | Provenance |
| --- | --- | --- | --- |

## Warnings
```

## Incremental Semantics

- File changes MUST update the owning `code_file` concept, all embedded symbol records, affected relation rows, and any generated repository/package/context views that reference them.
- Deleted files MUST remove stale generated file/symbol/relation records but MUST NOT delete human-authored concepts unless they carry explicit generated metadata.
- Renames SHOULD be represented as delete + create in the first implementation unless Git rename detection is already available.
- If state is missing or incompatible, OKF MUST fall back to full regeneration of generated code dimension artifacts.

## Risks / Trade-offs

- [Risk] Too many standalone symbol concepts can flood `.okf/knowledge`. → Mitigation: first version embeds symbols in file concepts and only emits standalone symbols for exported/public or configured kinds.
- [Risk] CodeGraph and OKF extractors disagree on symbol boundaries. → Mitigation: store provenance and keep OKF stable identity independent of CodeGraph ids.
- [Risk] Generated relation indexes become stale after incremental update. → Mitigation: treat relation indexes and context views as derived artifacts rebuilt from the normalized entity/relation set.
- [Risk] Frontmatter custom fields may become too large. → Mitigation: keep large relation tables in generated body or separate index concepts; frontmatter stores identity, schema, provenance, and compact metadata only.

## Migration Plan

1. Add internal code dimension model and renderer tests without changing default CLI behavior.
2. Convert existing file summaries and symbols into `CodeEntity`/`CodeRelation` during generation.
3. Render code-specific frontmatter custom fields and stable Markdown sections for `code_file` and repository index concepts.
4. Extend update flow to reconcile generated code entities and rebuild derived views.
5. Add query filters for symbol kind, language, file path, relation kind, and external id.
6. Add optional CodeGraph-compatible import format and mapping tests.
7. Validate with full regeneration vs incremental convergence tests.

## Open Questions

- 是否默认生成 standalone `code_symbol` concepts？建议第一版只为 exported/public symbols 或通过配置开启。
- CodeGraph 兼容导入的首个格式应是 JSON export 还是直接 SQLite 读取？建议先定义 JSON-like records，避免第一版绑定外部 DB schema。
- `code_context_view` 是每次 update 自动生成，还是按 CLI 参数生成？建议 repository overview 和 relation index 自动生成，任务特定 context view 按需生成。
